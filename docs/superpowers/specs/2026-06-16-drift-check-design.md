# drift_check Design

## Goal

Add a read-only `drift_check.yml` playbook that renders every config file the
`npm_ha` role manages — using the current repo variables — and diffs each against
the live file on the nodes, printing the diffs and exiting non-zero if anything
drifts. Run it **before** `main.yml` to catch repo↔live divergence (like the
2026-06-16 DRBD resource-name rename) before a converge can act on it. `main.yml`
is not modified.

## Context

On 2026-06-16, re-running `main.yml` against the live MIBTECH cluster caused a
production outage: a repo rename of `drbd_resource_app` re-rendered
`/etc/drbd.d/npm-ha.res` with a name pacemaker could not find, stopping DRBD and
the npm_group (VIP + NPM). The repo had silently drifted from the deployed state
and nothing surfaced it before the destructive run. `drift_check` is the guardrail
that would have caught it.

The `npm_ha` role manages these files (exact task params, which the drift check
must mirror so it does not report false owner/mode drift):

| File | Module | Source / params | Defined in |
|---|---|---|---|
| `/etc/drbd.d/npm-ha.res` | `template` | `npm-ha.res.j2`, root:root, `0600` | `tasks/drbd.yml` |
| `{{ npm_compose_dir }}/docker-compose.yml` | `template` | `docker-compose.yml.j2`, root:root, `0600` | `tasks/prepare_app.yml` |
| `/etc/systemd/system/npm-stack.service` | `copy: content:` | inline, root:root, `0644`, notify Reload systemd | `tasks/prepare_app.yml` |
| `/etc/hosts` node entries | `lineinfile` | line `"{{ip}} {{hostname}}"`, regexp `^[0-9.]+\s+{{hostname}}(\s.*)?$`, looped over both nodes | `tasks/prepare.yml` |
| corosync cluster name | `command: grep` | `pacemaker_cluster_name` present in `/etc/corosync/corosync.conf` | `tasks/cluster.yml` |

`/etc/corosync/corosync.conf` is generated by `pcs`, not templated by the role, so
its check is a **presence** sanity check (cluster name still there), not a
render-diff.

## Architecture

```
drift_check.yml                         — CREATE: playbook, hosts ha_nodes, become, read-only
roles/npm_ha/
  tasks/drift_check.yml                 — CREATE: per-file check_mode+diff tasks + final assert
  templates/npm-stack.service.j2        — CREATE: extracted systemd unit (DRY refactor)
  tasks/prepare_app.yml                 — MODIFY: systemd unit copy->template using the new .j2
```

`drift_check.yml` does `import_role: { name: npm_ha, tasks_from: drift_check }`.
Placing the tasks **inside** the `npm_ha` role is what makes the check DRY: it
resolves the role's own `templates/*.j2`, so it diffs the *exact* templates
`main.yml` would push — not copies that could drift from the originals.

### DRY refactor: extract the systemd unit to a template

The systemd unit is currently inlined as `copy: content:` in `prepare_app.yml`. To
drift-check it against the real source without duplicating that content, extract it
to `roles/npm_ha/templates/npm-stack.service.j2` and have **both** `prepare_app.yml`
and `drift_check.yml` render that one template. The rendered bytes are identical
(only `{{ npm_compose_dir }}` is interpolated), so the change causes no drift
itself; it removes the duplication that would otherwise let the check go stale.

The `.j2` must end with exactly one trailing newline to match the existing
`content: |` block output, avoiding a false-positive trailing-newline diff on the
first check.

## Data flow

`drift_check.yml` runs on both `ha_nodes` with `become: true` (the `.res` and unit
files are root-readable only). For each managed file, one task runs the **same
module** `main.yml` uses, with `check_mode: true` and `diff: true`, and
`register`s the result. `check_mode` guarantees no write; `diff: true` prints the
unified diff of what *would* change regardless of CLI flags.

After all checks, a `set_fact` builds `drift_check_drifted` — the list of files
whose check reported `changed` — and a final `assert` fails when that list is
non-empty, naming each drifted file. Diffs for those files have already printed
above the failure.

```yaml
# shape of the final gate
- name: Collect drifted files
  ansible.builtin.set_fact:
    drift_check_drifted: >-
      {{ (['/etc/drbd.d/npm-ha.res'] if drift_res.changed else [])
       + ([npm_compose_dir ~ '/docker-compose.yml'] if drift_compose.changed else [])
       + (['/etc/systemd/system/npm-stack.service'] if drift_unit.changed else [])
       + (['/etc/hosts'] if drift_hosts.changed else [])
       + (['corosync:' ~ pacemaker_cluster_name] if drift_corosync.rc != 0 else []) }}

- name: Assert no drift
  ansible.builtin.assert:
    that: drift_check_drifted | length == 0
    fail_msg: >-
      DRIFT DETECTED in {{ drift_check_drifted | length }} item(s):
      {{ drift_check_drifted | join(', ') }}.
      Review the diffs above and reconcile repo vs live before running main.yml.
    success_msg: "No drift: all managed npm_ha config matches the repo."
```

Each check task sets `changed_when` appropriately: the `command: grep` for corosync
uses `changed_when: false` and `failed_when: false` (so a missing name surfaces via
`rc`, not a task failure).

## Error handling / safety

- **Read-only by construction:** every check task is `check_mode: true`; the
  playbook cannot modify live state even if a node is in an unexpected condition.
- **No `any_errors_fatal`:** both nodes run fully and report their own diffs, so
  per-node drift is visible.
- **Gating-friendly:** drift makes the play exit non-zero (the `assert`), suitable
  for "stop before `main.yml`" use.

## Testing / acceptance

No unit test — the behaviour is pure live-node interaction. Acceptance, in order:

1. `ansible-playbook drift_check.yml` now → must **PASS** ("No drift"), since
   repo↔live were reconciled on 2026-06-16.
2. Temporarily change one var (e.g. `container_timezone`) → `drift_check.yml` must
   **FAIL**, printing the docker-compose diff and naming the file. Revert the var.
3. `ansible-playbook main.yml --check --diff --tags app` is NOT used as the
   mechanism (check mode misfires on the role's `pcs`/`drbd` command tasks); the
   dedicated `drift_check` tasks avoid those entirely.

## Out of scope

- Auto-gating `main.yml` (chosen: standalone playbook; `main.yml` unchanged).
- Diffing data that is not template-rendered by the role (DRBD on-disk contents,
  pacemaker resource definitions, the database).
- Auto-remediation — `drift_check` only reports; the human decides whether to fix
  the repo or the live node.
